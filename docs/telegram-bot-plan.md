# תוכנית: תמיכה ב-Telegram Bot לתמלול סרטונים

> מסמך תכנון בלבד — **טרם מומש**. מטרתו להסביר איך זה יעבוד, מה האילוצים, ומה אני ממליץ.
> תאריך: 18/07/2026

---

## 1. מה אתה רוצה (סיכום הדרישה)

1. שולח סרטון לבוט בטלגרם → הבוט **מוריד** אותו, **שומר** בתיקיית הווידאו, ו**מתחיל תמלול** אוטומטית.
2. בסיום — הבוט **מודיע** לך מה הסתיים ומה נכשל.
3. **פקודות** לשאול איפה הדברים עומדים ולבקש את התמלול ("לך תביא לי את התמלול", סטטוס, רשימה וכו').
4. **הגדרה מלאה דרך הממשק** — מזינים את כל הקודים (token, api_id, api_hash, ה-user-id שלך) פעם אחת ב-localhost, עם הסברים מה לעשות, וזה **נשמר לתמיד** גם אחרי כיבוי/הדלקה של הקונטיינרים. (ר' §3.5 — זו דרישה מרכזית.)
5. **שינוי שם קבצים** — סרטונים מהטלפון מגיעים עם שמות מוזרים (`IMG_1865.MOV`, `video_2026….mp4`), אז הבוט צריך לדעת לתת/לשנות שם, עם פקודות סלאש ייעודיות. (ר' §6.5.)

הכל צריך להשתלב במערכת הקיימת (FastAPI + Celery + Redis + SQLite + faster-whisper) בלי לשבור אותה.

---

## 2. ⚠️ האילוץ הקריטי — מגבלת 20MB (חובה לקרוא)

זו הנקודה שמכתיבה את כל הארכיטקטורה:

| | Bot API רגיל (ענן) | **Local Bot API Server** (self-hosted) |
|---|---|---|
| הורדת קובץ ע"י בוט | **20MB בלבד** | **עד 2GB** |
| איך מקבלים את הקובץ | URL להורדה דרך HTTP | **נתיב מקומי על הדיסק** (בלי הורדה בכלל) |
| מה צריך | טוקן מ-@BotFather | טוקן + `api_id`/`api_hash` מ-my.telegram.org |

**המשמעות:** 20MB זה ~1–2 דקות וידאו באיכות רגילה. סרטון הרצאה של 40 דקות יכול להיות 200–800MB. לכן, כדי שהבוט יהיה **שימושי באמת**, חייבים להריץ **Local Bot API Server** — קונטיינר נוסף שרץ אצלך, שמאפשר עד 2GB ומחזיר את הקובץ כ**נתיב מקומי** (הבוט פשוט מעתיק אותו לתיקיית הווידאו, בלי הורדת HTTP כלל).

זה נשמע מפחיד אבל זה עוד קונטיינר אחד ב-docker-compose. החיסרון היחיד: צריך `api_id`/`api_hash` (חינמי, דקה של רישום ב-my.telegram.org).

> **חלופה "קלה":** אם היית מוכן להסתפק בסרטונים ≤20MB — אפשר בלי השרת המקומי. אבל לדעתי זה לא ריאלי לסרטונים אמיתיים, אז ההמלצה שלי היא ללכת על Local Bot API Server מההתחלה.

## 3. ארכיטקטורה מוצעת

מוסיפים **שני** שירותים ל-docker-compose: `telegram-bot-api` (השרת המקומי) ו-`bot` (הלוגיקה שלנו).

```
   הטלפון שלך                                  המחשב (Docker)
  ┌──────────┐   שולח וידאו   ┌────────────────────────┐
  │ Telegram │ ─────────────► │  telegram-bot-api      │  (שרת מקומי, עד 2GB)
  │  client  │ ◄───────────── │  שומר את הקובץ לדיסק     │
  └──────────┘   הודעות/סטטוס  └───────────┬────────────┘
                                           │ getUpdates (long polling)
                                           │ getFile → נתיב מקומי
                                 ┌─────────▼─────────┐
                                 │   bot (חדש)        │ python-telegram-bot
                                 │  • מזהה משתמש מורשה │
                                 │  • מעתיק לתיקיית    │──► /app/videos
                                 │    הווידאו          │
                                 │  • יוצר job ב-DB    │──► SQLite (משותף)
                                 │  • שולח Celery task │──► Redis
                                 │  • מודיע בסיום      │
                                 └─────────┬─────────┘
                                           │ שולף סטטוס/התראות
        api (FastAPI) ◄── משותף ──► SQLite ◄──┘   worker (Celery) ──► SRT/TXT
```

**נקודות מפתח:**
- ה-`bot` חולק את אותו **SQLite** ואת אותו **Redis** כמו ה-api וה-worker — אז הוא רואה בדיוק את אותם jobs, ויכול ליצור jobs חדשים בדיוק כמו שהממשק עושה.
- ה-`bot` צריך **גישת כתיבה** לתיקיית הווידאו (כדי לשמור סרטונים). כרגע היא ממופה `:ro` (read-only) ל-api ול-worker — ל-bot נמפה אותה **read-write**.
- **Long polling** (getUpdates), לא webhook — אין צורך ב-URL ציבורי או HTTPS. מושלם למערכת אישית מקומית.

## 3.5 הגדרה מלאה דרך הממשק + שמירה קבועה  ⭐ (דרישה מרכזית)

**המטרה:** להגדיר את כל חיבור הטלגרם **מתוך הממשק ב-localhost**, פעם אחת — בלי לערוך קבצי `.env` ידנית. אחרי שמירה, ההגדרות **נשמרות לתמיד**: גם אחרי `restart`, `docker compose down/up`, ואפילו rebuild.

### איפה זה נשמר (וזה מה שמבטיח קביעוּת)
- ההגדרות נשמרות בטבלה חדשה `settings` בתוך ה-**SQLite הקיים**, שכבר יושב על ה-volume הקבוע `./data`.
- volumes נשמרים בין הרצות (בדיוק כמו ה-whisper-cache) → מזינים פעם אחת, וזה שם לתמיד.
- ה-DB כבר ב-`.gitignore`, אז הסודות אף פעם לא עולים ל-git.

### דף ההגדרות בממשק (חדש)
כפתור **⚙️ הגדרות** בממשק פותח עמוד/חלון עם:
1. **הסברים שלב-אחר-שלב** (עם קישורים) — איך יוצרים בוט ב-[@BotFather](https://t.me/BotFather), איך משיגים `api_id`/`api_hash` ב-[my.telegram.org](https://my.telegram.org/auth), ואיך מוצאים את ה-user-id שלך (למשל דרך @userinfobot).
2. **שדות להזנה**: Bot Token · API ID · API Hash · Allowed user IDs · מתג הפעלה (On/Off).
3. **כפתור שמירה** → נשמר ל-DB מיד.
4. **חיווי חיבור חי**: `לא מוגדר · מתחבר… · מחובר ✅ · שגיאה ❌ (הסיבה)` — כדי שתדע מיד אם הטוקן תקין.
5. **"בדוק חיבור"** — מריץ getMe ומראה את שם הבוט.
6. סודות מוצגים ממוסכים (`•••••1234`) אחרי שמירה, עם אפשרות לעדכן.

### איך זה נכנס לתוקף בלי לגעת בקונטיינרים
- **שירות `bot`**: רץ תמיד, אבל "רדום" עד שמוגדר. לולאת-פיקוח קטנה קוראת את `settings` מה-DB; כשמופעל ותקין → מעלה את חיבור הטלגרם (long polling); כשמשנים token/allowed-ids/הפעלה → הוא **טוען מחדש לבד**, בלי restart לקונטיינר. הוא גם כותב את **סטטוס החיבור** בחזרה ל-DB, וזה מה שהממשק מציג.
- **שירות `telegram-bot-api`** (השרת המקומי): הוא צריך את `api_id`/`api_hash` כבר בעלייה. לכן ה-entrypoint שלו **ממתין** לקובץ הגדרות על `./data` (שהממשק כותב), קורא ממנו, ואז עולה. בהפעלה ראשונה — ברגע שתשמור בממשק, השרת מתחיל לבד. גם אחרי כיבוי/הדלקה הוא פשוט קורא שוב את אותו קובץ.
  - **דקוּת יחידה שכדאי להכיר:** אם **תשנה** את ה-`api_id`/`api_hash` בעתיד (נדיר — עושים פעם אחת), השרת המקומי צריך `docker compose restart telegram-bot-api` כדי לקלוט. כל השאר (token, allowed-ids, הפעלה/כיבוי) נטען חם בלי restart.

### טבלת קביעוּת — מה נשמר ואיפה
| מה | נשמר ב | שורד restart / down-up / rebuild? | טעינה חמה? |
|---|---|---|---|
| Bot Token, Allowed IDs, הפעלה | טבלת `settings` (SQLite על `./data`) | ✅ | ✅ (הבוט טוען לבד) |
| api_id / api_hash | `settings` + קובץ על `./data` | ✅ | ⚠️ שינוי דורש restart לשרת המקומי בלבד |
| סטטוס חיבור (לתצוגה) | טבלת `settings` | — | ✅ |

### אבטחה
הסודות נשמרים כטקסט מקומי ב-SQLite שלך (אותה רמת אמון כמו `.env`) — לא עוזבים את המחשב, וה-DB ב-`.gitignore`. למערכת אישית מקומית זה בסדר גמור. (אם נרצה בעתיד — אפשר להוסיף הצפנה בכספת/סיסמת-אב, אבל זה overkill כרגע.)

## 4. שינויים נדרשים בקוד הקיים

### 4.1 טבלת `jobs` (הרחבה, לא שבירה — עם מיגרציה כמו שעשינו ל-progress)
מוסיפים עמודות:
- `source` — `web` / `telegram` (מאיפה ה-job הגיע)
- `chat_id` — למי להודיע בסיום (ה-chat של טלגרם)
- `notified` — האם כבר נשלחה התראת סיום (למניעת התראות כפולות)

### 4.2 טבלה חדשה `settings` (key/value) — לשמירת הגדרות הטלגרם מהממשק
מחזיקה את: `bot_token`, `api_id`, `api_hash`, `allowed_ids`, `enabled`, וגם `connection_status`/`last_error` שהבוט כותב בחזרה לתצוגה. יושבת ב-SQLite הקיים (persistent).

### 4.3 שירותים חדשים ב-`docker-compose.yml`
- `telegram-bot-api` — image רשמי (`aiogram/telegram-bot-api` או `tdlib/telegram-bot-api`), עם volume משותף לקבצים ו-**entrypoint שקורא api_id/api_hash מקובץ על `./data`**.
- `bot` — נבנה מאותו Dockerfile, מריץ `python app/bot.py` עם לולאת-פיקוח שקוראת הגדרות מה-DB.

### 4.4 קבצים/קוד חדשים
- `app/bot.py` — כל לוגיקת הבוט (handlers, פקודות, התראות, לולאת reload). לא נוגע ב-worker/api.
- `app/main.py` — endpoints חדשים: `GET /api/settings/telegram` (מחזיר הגדרות עם סודות ממוסכים + סטטוס) ו-`POST /api/settings/telegram` (שמירה).
- `frontend/` — דף/חלון **הגדרות** עם ההסברים, השדות, וחיווי החיבור (ר' §3.5).
- `scripts/telegram-api-entrypoint.sh` — entrypoint לשרת המקומי שממתין לקובץ ההגדרות ואז מפעיל אותו.

### 4.5 תלות חדשה ב-`requirements.txt`
`python-telegram-bot` (גרסה 22.x — v22.6 נכון ליולי 2026, אסינכרוני, Python 3.10+).

## 5. פקודות הבוט (הצעה)

| פקודה | מה עושה |
|---|---|
| *(שליחת סרטון)* | מוריד, שומר לתיקייה, יוצר job, מתחיל תמלול, מאשר לך |
| `/start` · `/help` | הסבר קצר + רשימת פקודות |
| `/status` | מצב התור: מה בעיבוד (עם **אחוז**), מה בתור, כמה הושלמו |
| `/list` | רשימת כל הסרטונים והסטטוס שלהם |
| `/get <שם או מספר>` | שולח לך בחזרה את קבצי ה-SRT + TXT של תמלול שהושלם |
| `/last` | שולח את התמלול האחרון שהסתיים |
| `/all` | מפעיל תמלול על כל הסרטונים שעדיין לא תומללו (כמו "עבד את הכל") |
| `/rename <מספר#> <שם חדש>` | משנה שם לסרטון (וגם לתמלולים שלו) לפי מספר |
| `/rename <שם נוכחי> <שם חדש>` | אותו דבר, לפי השם הנוכחי |
| `/name <שם חדש>` | בתגובה (reply) להודעת "התקבל" — נותן שם לסרטון שהרגע נשלח |
| `/cancel <מספר>` | *(אופציונלי)* מבטל job שממתין בתור |

> "לך תביא לי את התמלול" = `/get` או `/last`. אפשר גם לתמוך בטקסט חופשי בסיסי, אבל פקודות `/` יותר אמינות וברורות.

## 6. זרימות עבודה מרכזיות

**שליחת סרטון:**
1. אתה שולח וידאו (עדיף **כ"קובץ/Document"** ולא "וידאו" — שומר על האיכות המקורית בלי דחיסה של טלגרם).
2. הבוט בודק שאתה **משתמש מורשה** (ראה אבטחה למטה).
3. **קביעת שם**: אם צירפת **כיתוב (caption)** להודעה — הוא הופך לשם הקובץ. אחרת נשמר בשם המקורי מהטלפון (שלרוב "מוזר", למשל `IMG_1865.MOV`) — ותוכל לשנות אח"כ (ר' §6.5).
4. `getFile` מחזיר נתיב מקומי → הבוט מעתיק ל-`/app/videos/<שם נקי>.<סיומת>` (מטפל בשמות כפולים ובתווים בעייתיים, שומר את הסיומת המקורית).
5. הבוט יוצר job ב-DB (`source=telegram`, `chat_id=...`) ושולח Celery task — בדיוק כמו הכפתור בממשק.
6. הבוט עונה: "📥 התקבל `IMG_1865.MOV` · נוסף לתור (#7). לשינוי שם: השב `/name <שם>`".

**התראת סיום (מה הסתיים / מה נכשל):**
- ה-`bot` בודק את ה-DB כל כמה שניות (polling), ומחפש jobs שהסתיימו (`done`/`failed`) ועדיין `notified=0`.
- לכל אחד הוא שולח הודעה ומסמן `notified=1`:
  - ✅ "`lecture.mp4` הושלם · [SRT] [TXT]"
  - ❌ "`broken.mp4` נכשל · הסיבה: ..."
- **למה polling ולא push?** כי זה עמיד: גם אם הבוט היה כבוי בזמן הסיום, ברגע שהוא עולה הוא יראה את מה שפספס וישלח. (אלטרנטיבה: Redis pub/sub — אלגנטי יותר אבל מאבד הודעות אם הבוט למטה. לכן polling+flag מנצח כאן.)

**קבלת תמלול (`/get`, `/last`):**
- הבוט שולף מה-DB את נתיבי ה-SRT/TXT ושולח אותם כקבצים בטלגרם.

### 6.5 שינוי שם קבצים ⭐

**למה צריך:** סרטוני טלפון מגיעים עם שמות לא-קריאים (`IMG_1865.MOV`, `video_2026-07-18_....mp4`). כדי שיהיה קל למצוא ולהוריד תמלולים, רוצים לתת להם שמות אנושיים.

**שלוש דרכים לתת שם (מהנוח לגמיש):**
1. **כיתוב בזמן שליחה** — שולחים את הסרטון עם caption "שיעור תנ״ך 3" → זה השם.
2. **תשובה מיד אחרי שליחה** — `/name שיעור תנ״ך 3` בתגובה להודעת "התקבל".
3. **פקודת rename מתי שרוצים** — `/rename 7 שיעור תנ״ך 3` (לפי מספר) או `/rename IMG_1865 שיעור תנ״ך 3` (לפי שם נוכחי).

**מה שינוי שם עושה בפועל (אטומי — הכל-או-כלום):**
- משנה את קובץ הווידאו: `videos/IMG_1865.MOV → videos/שיעור-תנך-3.MOV` (**הסיומת נשמרת אוטומטית**).
- אם כבר תומלל — משנה גם את הפלטים: `transcripts/IMG_1865.srt/.txt → transcripts/שיעור-תנך-3.srt/.txt`.
- מעדכן את רשומת ה-job ב-DB (`filename`, `srt_path`, `txt_path`) — כך שגם **הממשק** מראה את השם החדש מיד.

**כללים וקצוות:**
- **ניקוי שם (sanitize)**: רווחים→מקפים, הסרת תווים בעייתיים ומניעת path traversal (בלי `/` או `..`).
- **שם תפוס** → הבוט מציע חלופה או מבקש שם אחר (לא דורס בשקט).
- **חסום בזמן `processing`**: אי אפשר לשנות שם לסרטון שמתומלל כרגע (כדי לא לשבור את ה-job הרץ) — הבוט יבקש לחכות לסיום.
- הבוט מאשר: "✏️ שונה ל-`שיעור-תנך-3.MOV`".

## 7. אבטחה (חשוב — הבוט מוריד ומריץ על המחשב שלך)

- **Allowlist**: רשימת ה-user-ids המורשים מוגדרת ב**דף ההגדרות בממשק** (§3.5) ונשמרת ב-DB. כל הודעה ממשתמש אחר — נדחית בשקט. **חובה**, אחרת כל אחד שמוצא את הבוט יכול להעלות קבצים למחשב שלך.
- הטוקן וה-`api_id`/`api_hash` נשמרים ב-SQLite המקומי (על `./data`, שכבר ב-`.gitignore`) — לא עוזבים את המחשב, אף פעם לא ב-git.
- ולידציה על סוג הקובץ (רק סיומות וידאו מותרות) ועל הגודל.
- שמות קבצים "מנוקים" (sanitize) כדי למנוע path traversal.

## 8. מה צריך ממך (חד-פעמי — הכל דרך הממשק)

את שלושת אלה משיגים פעם אחת ומזינים ב**דף ההגדרות בממשק** (לא בקבצים), ואז זה נשמר לתמיד (§3.5):

1. **Bot token** — יצירת בוט אצל [@BotFather](https://t.me/BotFather) → `/newbot`.
2. **api_id + api_hash** — רישום ב-[my.telegram.org](https://my.telegram.org/auth) (חינמי) — נדרש ל-Local Bot API Server.
3. **ה-user-id שלך** — כדי להגדיר את ה-allowlist (אפשר לקבל מבוט כמו @userinfobot).

הממשק ילווה אותך שלב-אחר-שלב איך להשיג כל אחד. בלי (2) הבוט יעבוד רק לקבצים ≤20MB. בלי (1) ו-(3) הוא לא יעבוד/לא יהיה מאובטח.

---

## 9. תוכנית שלבים (כשנתחיל)

1. **שלד**: הוספת `bot` + `telegram-bot-api` ל-compose, `bot.py` בסיסי שעונה ל-`/start` ומזהה משתמש מורשה. אימות שהחיבור (כולל local server) עובד.
2. **קליטת וידאו**: שליחת סרטון → שמירה לתיקייה → יצירת job → הפעלת תמלול. (מיגרציית ה-DB כאן.)
3. **התראות**: לולאת ה-polling על ה-DB + הודעות סיום (done/failed).
4. **פקודות שליפה**: `/status`, `/list`, `/get`, `/last`, `/all`.
5. **ליטוש**: הודעות יפות, טיפול בקצוות (קבצים כפולים, פורמטים לא נתמכים, קבצים ענקיים), עדכון README.
6. **בדיקה end-to-end** מהטלפון, כמו שעשינו עם הממשק.

**הערכת מאמץ:** ליבה עובדת (שלבים 1–3) — סשן עבודה ממוקד אחד. הכל כולל ליטוש — סשן-וחצי בערך. רוב ה"סיכון" הוא ההגדרה החד-פעמית של ה-local API server, לא הקוד.

## 10. מה אני חושב (המלצה כנה)

**זה רעיון מצוין ומתאים בול למערכת הזו.** התשתית שכבר בנינו (תור אסינכרוני, jobs ב-DB, error-handling per-job) הופכת את זה כמעט ל"פלאג-אין": הבוט הוא עוד **מקור** ל-jobs לצד הממשק, ומשתמש באותו worker בדיוק. לא צריך לגעת בלוגיקת התמלול.

**מה שהייתי שם עליו דגש:**
0. **הגדרה דרך הממשק (§3.5) היא הבחירה הנכונה** — הרבה יותר נוח מ-`.env`, וה-SQLite על ה-volume הקבוע נותן את ה"פעם-אחת-ונשמר" בחינם. הדקוּת היחידה: api_id/api_hash לשרת המקומי נטענים ב-entrypoint, ושינוי שלהם (נדיר) דורש restart לשירות אחד. שאר ההגדרות נטענות חם. שווה את זה.
1. **ה-Local Bot API Server הוא לא אופציונלי בפועל** — בלעדיו מגבלת 20MB הופכת את הבוט לצעצוע. שווה את ההגדרה החד-פעמית. זו ההחלטה הכי חשובה כאן.
2. **תשלח סרטונים כ"קובץ/Document"** ולא כ"וידאו" — אחרת טלגרם דוחס ואיכות האודיו (וממילא התמלול) נפגעת.
3. **אבטחה (allowlist) היא חובה מהיום הראשון**, כי הבוט כותב ומריץ על המחשב האישי שלך.
4. **התראות ב-polling+flag** (ולא pub/sub) — פחות "מגניב", אבל עמיד לכיבוי/קריסה של הבוט, וזה מה שחשוב למערכת "תזרוק לילה שלם".

**מה הייתי דוחה לגרסה 2:** ביטול jobs, ניהול תור מתקדם, טקסט חופשי חכם (NLP) במקום פקודות `/`. לא נחוץ להתחלה.

**סיכון/חיסרון עיקרי:** התלות ב-`api_id`/`api_hash` וההגדרה של השרת המקומי — צעד חד-פעמי אבל לא לגמרי טריוויאלי. חוץ מזה, אין פה סיכון ארכיטקטוני אמיתי.

**שורה תחתונה:** ממליץ לממש, עם Local Bot API Server, בגישה מדורגת (שלבים 1–3 קודם, לראות שזה עובד מהטלפון, ואז פקודות וליטוש).
